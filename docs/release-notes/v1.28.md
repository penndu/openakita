# OpenAkita v1.28 — Release Notes

> Status: pre-release notes for the v1.28 family
> (boot-fault-tolerance v1.29 plan + setup-flow v1.28 plan).

This file collects user-visible changes that ship in v1.28 series.
Lower-level technical detail lives in the per-plan documents in
`.cursor/plans/`.

## Boot fault tolerance (Stage 1 + Stage 2 + Stage 3)

OpenAkita now treats a corrupted SQLite database or half-written JSON
state file as a degraded condition instead of a startup failure. The
new pipeline:

1. Every long-lived SQLite open goes through `safe_sqlite.safe_open_*`,
   which runs `PRAGMA quick_check`, detects hot-journal orphans, and
   surfaces a structured `SQLiteUnavailable(reason=...)` exception.
2. Subsystems that can't open their DB (`memory`, `token_tracking`,
   `feedback`, `asset_bus`, `database`) register themselves in
   `openakita.storage.degraded.registry`. The user sees a yellow banner
   in the desktop app and `/api/health` reports
   `degraded_subsystems: [...]`.
3. From the banner the user can choose to "quarantine + recreate"
   each subsystem: the helper closes all file handles, renames the
   `.db` + `-wal` + `-shm` triplet into `data/.quarantine.{ts}/`, and
   reports `restart_required=true`. No user data is silently deleted.
4. All JSON state files under `data/` now use atomic temp+rename with
   `.bak` backup and a `read_json_safe` fallback. A kill -9 mid-write
   can no longer leave behind an unparseable JSON that blocks startup.

## Breaking change for plugin authors: `AssetBus` returns empty/None when degraded

Before v1.28, `assets_bus` operations raised when the underlying SQLite
file was unreadable. As of v1.28 the bus degrades silently:

* `bus.publish(...)` returns a placeholder asset_id but no row is
  persisted.
* `bus.get(...)` returns `None`.
* `bus.list_owned(...)` returns `[]`.
* `bus.delete_owned`, `sweep_expired`, `sweep_owner`, `count_all` all
  short-circuit to `0` / `False`.

Plugins need to tolerate `None` / `[]` from these calls and check
`GET /api/memory/repair/degraded` to detect bus degradation. Example:

```python
asset = await bus.fetch(asset_id, requester_plugin_id="my-plugin")
if asset is None:
    return None  # or fall back to your own cache
preview_url = asset.preview_url
```

The bus also exposes `async def quiesce()` which the host calls before
quarantining `data/asset_bus.db`. Plugins that hold a reference to the
bus instance should not rely on it being usable across a quiesce.

## New endpoints

* `GET  /api/memory/repair/degraded` — read-only snapshot of the
  module-level `DegradedRegistry` plus a fresh confirmation token.
* `POST /api/memory/repair/quarantine` — body
  `{subsystem, confirmation_token}` (header
  `X-OpenAkita-Desktop-Token` required in production). Moves the
  subsystem's DB triplet to `data/.quarantine.{ts}/` and clears the
  degraded flag.
* `GET  /api/health` payload now includes `degraded_subsystems: [...]`.

## Deprecations

* `openakita.storage.database.Database` is marked legacy and emits a
  `DeprecationWarning` at construction. New code should route through
  the subsystem-specific store (`MemoryStorage`, `AssetBus`,
  `feedback_store`, …) which already use `safe_sqlite` underneath.
  The class itself is now safe to keep using; the warning exists to
  steer future contributors.

## v1.27.14 hotfix — Conversation concurrency Stage 1 (plan v1.28)

The post-`v1.27.12` crash report (`issue #572` —
`completed -> reasoning` state-machine violation) revealed that a single
`TaskState` was shared between concurrent requests on the same
`conversation_id`. Stage 1 of the conversation-concurrency plan
hardens this surface end-to-end **without** changing default user
behavior on the desktop client. See
`.cursor/plans/conversation_concurrency_v1.28_*.plan.md` for the full
multi-stage design (S1–S5).

### New double-texting policy framework

* New enum `openakita.api.routes.double_texting.DoubleTextingPolicy`
  with values `REJECT` / `QUEUE` / `INTERRUPT` / `STEER`.
* New helper `resolve_policy(channel=..., header_value=...)` picks the
  policy from:
  1. HTTP header `X-OpenAkita-DoubleTexting` (per-request override)
  2. `settings.double_texting_per_channel[channel]`
  3. `settings.double_texting_default`
* Conservative defaults shipped:
  * `double_texting_default = "queue"`.
  * Per-channel defaults: IM channels (feishu / wework / dingtalk /
    wechat) default to `reject`; desktop / cli / telegram / qqbot /
    onebot default to `queue`.
  * **`double_texting_allow_interrupt = false`** — any caller asking
    for `INTERRUPT` is silently down-graded to `QUEUE` until S4
    (v1.28.2) lands tool `interrupt_behavior` metadata. This is the
    safe default that protects long-running `run_shell` / browser
    actions from being torn down mid-flight.
* `preempt_settle_timeout_ms = 6000` controls how long a new request
  waits for the previous task to settle before marking it
  `abandoned` and proceeding.

### Conversation lifecycle is now per-policy

`ConversationLifecycleManager.start()` returns a richer `StartResult`
(carrying `conflict`, `generation`, `took_over`, `policy_applied`,
`queued_after_generation`). Same-client overlaps now produce four
distinct outcomes instead of a silent overwrite:

* `REJECT` → 409 with `conversation_busy`.
* `QUEUE` → caller waits for the prior task's `settled_event`.
* `INTERRUPT` / `STEER` → previous task is cancelled, new lock issued,
  broadcast carries `took_over_generation` for the frontend banner.

`BusyInfo` grew a `turn_id` field so admin diagnostics
(`GET /api/chat/busy`) report which client+turn currently owns the
slot. `StartResult` keeps tuple-unpacking back-compat
(`conflict, gen = await start(...)`) for callers that haven't migrated.

### Per-turn idempotency (`turn_id`)

* `ChatRequest.turn_id` is an optional `[A-Za-z0-9_\-:.@]{0,128}`
  identifier the client supplies for a single user turn. SSE reconnects
  / network retries should reuse the same `turn_id`; a brand-new send
  should use a fresh value.
* In-process `TurnRegistry` (`api.routes.turn_registry`) short-circuits
  duplicates: `in_flight` → `409 turn_already_processing` (with
  `Retry-After: 2`); finished (`succeeded`/`failed`) →
  `409 turn_already_finished`. Records auto-expire after 60s.
* This is in-process only; multi-worker deployments are explicitly out
  of scope until v1.29. Workers must be pinned to a single
  conversation by the load-balancer.

### Preempt protocol on the agent layer

`Agent.chat_with_session()` and `Agent.chat_with_session_stream()` now
delegate previous-task resolution to
`Agent._preempt_or_queue_prev_task()`, which replaces the legacy
"`clear_skip` + `drain_user_inserts`" code path that silently shared
one `TaskState` between two concurrent requests. The new flow:

1. If no prev task or prev task is dead → proceed (existing behavior).
2. If prev task is live → apply `resolve_policy(channel=...)`:
   * `REJECT` (HTTP-layer escape-hatch) → reset, log warning, proceed.
   * `QUEUE` → `await wait_until_settled` then reset; on timeout mark
     `abandoned=True` and append a "上一条任务被中断" marker.
   * `INTERRUPT` / `STEER` → cancel prev, wait for settle, reset; on
     timeout mark `abandoned=True`. A stale `_pending_cancels` entry
     on this session is discarded to avoid killing the new task.

A new `TaskState.abandoned` flag short-circuits any straggler iteration
of the cancelled task so it can never write back into shared state.

### `TaskState.settled_event` / `mark_settled()`

`TaskState` learnt `settled_event: asyncio.Event` plus
`mark_settled()` / `wait_until_settled()`. The reasoning engine wraps
both `run()` and `reason_stream()` in an outer `try/finally` that
guarantees `mark_settled()` runs on every exit path (success,
exception, generator close, abandoned). The 5000-line bodies were
renamed to `_run_impl` / `_reason_stream_impl` and the public methods
became thin wrappers — call-sites are unchanged.

### Cancel marker on session history

`SessionContext.append_marker()` (and `Session.append_marker()`)
appends a record to `session.context.messages` **bypassing the
deduplication path** used by `add_message`. Preempt / abandon paths
now leave a visible `"[上一条任务被新请求中断]"` marker with
`marker_type=preempted`, `policy`, `preempted_task_id`, `reason`
metadata so the frontend timeline and the next LLM turn both know the
old answer was cut short.

### Telemetry

`openakita.core.conversation_metrics` exposes five counters with
`policy` / `channel` labels for grafana / log aggregation:

* `preempt` — INTERRUPT/STEER actually fired.
* `queue` — QUEUE wait taken.
* `settled_timeout` — `wait_until_settled` timed out.
* `abandon` — old task marked `abandoned` after timeout.
* `takeover` — HTTP `lifecycle.start` reported `took_over`.

Plus `illegal_reasoning_entry` reserved for S5 strict-mode wiring.

### Tests

* `tests/integration/test_conversation_concurrency.py` — 45 cases
  covering policy resolution, lifecycle policy-aware semantics
  (including `wait_for_idle` + QUEUE retry), `TaskState.settled_event`,
  the `_preempt_or_queue_prev_task` helper (conversation_id key + QUEUE
  timeout cancellation), cancel-marker dedup-bypass + persistence,
  `TurnRegistry`, and telemetry.
* Existing `tests/unit/test_reason_stream_state_race.py` (issue #572
  race-guard contract) was re-pointed at `_reason_stream_impl` and
  still passes — the v1.27.12 hotfix `try/except ValueError` blocks
  remain in place until S5 (v1.28.3) removes them in favour of the
  strict `IllegalReasoningEntry` path.

### Post-S1 follow-up patch (FIX 1–7, same release window)

The first cut of S1 shipped with several gaps caught by a follow-up
self-audit; all are addressed in this commit:

1. **Key mismatch** — `_preempt_or_queue_prev_task` now looks up the
   previous `TaskState` by `conversation_id` (the key
   `reason_stream` / `run` actually register under), falling back to
   `session_id` only for CLI / sub-agent callers that don't pass a
   `conversation_id`. Without this the preempt protocol silently
   missed the live task on IM / org-mode paths.
2. **Policy wiring** — `/api/chat` (SSE) now passes
   `policy=resolve_policy(channel="desktop", header_value=X-OpenAkita-DoubleTexting)`
   to `ConversationLifecycleManager.start()`. Previously the default
   was `REJECT`, so the QUEUE default in
   `settings.double_texting_per_channel` was effectively ignored and
   same-client double-texts still returned 409.
3. **`wait_for_idle` + QUEUE retry** — `ConversationLifecycleManager`
   gained `wait_for_idle(conv_id, target_generation, timeout)`; on
   QUEUE same-client conflict `/api/chat` now awaits the previous
   generation to settle (bounded by `preempt_settle_timeout_ms`) and
   retries `start()` once before falling back to 409. With FIX 2 +
   FIX 3 the QUEUE policy is actually reachable end-to-end.
4. **QUEUE timeout cancels the old task** — when
   `wait_until_settled` times out, the helper now calls
   `prev_task.cancel(...)` in addition to setting `abandoned=True`.
   Long-running tools (shell / browser) listen on `cancel_event`, not
   `abandoned`, so without this they kept running and raced the new
   turn's tools for cwd / fs side effects.
5. **Cancel-marker persistence** — `Session.append_marker` now mirrors
   `add_message`'s best-effort `_write_turn_to_store` path
   (history_db_merge_v1, skip on `transient_for_llm`, swallow
   exceptions). Markers survive a backend restart instead of silently
   making the timeline look as if the cut-off answer finished cleanly.
6. **Diagnostics endpoint** — new read-only routes
   `GET /api/diagnostics/conversation_metrics` and
   `GET /api/diagnostics/turn_registry` expose the in-process
   counters and in-flight turn registry for ops tuning of
   `preempt_settle_timeout_ms` / per-channel policy.
7. **Test stub hardening** — the integration test helper now
   instantiates the real `Agent` via `__new__` instead of declaring a
   parallel `_AgentStub` class with manually-rebound methods, so new
   `self.*` calls inside the helper don't silently break the suite.
   FIX 1, 3, 4, 5 each get dedicated regression cases.

## Conversation concurrency v1.28 — S2 (v1.27.15, external-project follow-ups)

After S1 landed, we audited nine external projects (`new-api`,
`Sub2API`, `litellm` on the gateway side; `claude-code`, `hermes-agent`,
`langchain`, `openclaw`, `openfang`, `QwenPaw` on the runtime side)
for patterns we should adopt, mirror, or deliberately reject. Six
patterns made it into S2; two (Redis-backed TurnRegistry, Prometheus
in-flight gauge) were rejected as over-engineering for the single-host
deployment OpenAkita actually ships.

### P0-1: SSE event buffer + `/api/chat/resume` reconnect endpoint

C17 Phase B already shipped per-conversation `SSESession` ringbuffers
(maxlen=100, TTL=300s) with `Last-Event-ID` replay. **In practice this
was dead code after S1** — `chat()` runs `lifecycle.start()` first,
which 409s same-client reconnects or queues them indefinitely, so the
replay branch inside `_stream_chat` was unreachable. v1.27.15 adds a
read-only **`GET /api/chat/resume?conversation_id=…&since_seq=N`**
endpoint that:

* Does NOT touch the lifecycle (no busy-lock change, no generation
  bump) — it is strictly attach-to-existing.
* Flushes ringbuffer events with `seq > since_seq` immediately.
* Tail-polls the ringbuffer every 100ms; emits new events as the
  still-running ReAct loop writes them.
* Closes cleanly when the lifecycle reports idle, the client
  disconnects, or 15 min elapse without new events.

This is the path the desktop UI takes on Wi-Fi reconnect / VPN
re-handshake / corporate-proxy idle timeout. Multiple subscribers per
conversation are supported (open the app on two devices to follow the
same turn from both).

### P0-2: SSE delta coalescer (50ms / 2000-char window)

User feedback in v1.27.10–v1.27.12 ("回复内容多了之后流式输出非常卡顿") traces back
to React saturating on per-token `text_delta` re-renders. New
`core/sse_throttle.py::DeltaCoalescer` merges `text_delta` /
`chain_text` / `thinking_delta` / `reasoning_delta` into per-channel
buckets and flushes a single merged event whenever any of:

* The bucket accumulates ≥2000 chars (one sentence-sized paragraph).
* 50ms elapsed since this bucket's last flush.
* A non-delta event arrives (always flushes pending in insertion
  order — preserves `text_delta` → `tool_call_start` ordering).
* End-of-stream / abort.

Non-delta events bypass the buffer entirely (zero added latency for
`tool_call_start`, `ask_user`, `artifact`, `done`). The HTTP chunk
count drops by ~10–40× on typical long answers; per-frame React work
collapses accordingly. The contract is content-additive: front-end
consumers see the same event types they always did, just fewer of
them with longer `content`.

### P0-3: Partial assistant text persisted on abort (`aborted_partial` marker)

Previously a mid-answer cancel made the timeline read as if the model
never spoke. Now `TaskState` accumulates `partial_text` and
`partial_thinking` (each capped at 16k chars, `partial_truncated`
flag flips when the cap is hit). The `reason_stream` outer wrapper
appends every streamed `text_delta` / `thinking_delta` / `chain_text`
into the live `TaskState`. When `_preempt_or_queue_prev_task` cancels
or abandons the previous task it now writes **two** markers instead of
one:

1. `marker_type="preempted"` placeholder (existing, plus
   `has_partial=True/False` flag).
2. `marker_type="aborted_partial"` with the actual partial
   `content`, tagged by `partial_channel ∈ {"text","thinking"}`.

The user now sees "the answer crawled this far before I cut it off"
rather than "[interrupted]" with no context.

### P1-4: STEER policy actually implemented

Until v1.27.15 `STEER` was a `DoubleTextingPolicy` enum value with no
distinct lifecycle path — it behaved identically to `INTERRUPT`. The
plan calls for STEER to mean "let the old turn keep running, just
inject this new message into its working context". v1.27.15 wires:

* `StartResult.steered: bool` — set true when a same-client request
  arrives with `policy=STEER` while a turn is in flight.
  Lifecycle does NOT acquire the lock, does NOT bump generation, does
  NOT broadcast `chat:busy`.
* `/api/chat` short-circuits on `steered=True`: calls
  `Agent.insert_user_message(message, session_id=conversation_id)`
  (existing facility — appends to `TaskState.pending_user_inserts`
  which the ReAct loop drains on the next post-tool tick) and returns
  HTTP `202 { status: "steered", resume_hint: { endpoint: "/api/chat/resume", … } }`.
* Front-end is expected to ack and re-attach via `/api/chat/resume`
  to continue receiving the (now-augmented) original turn.

STEER is OFF by default (`settings.double_texting_allow_interrupt=false`
downgrades `INTERRUPT` and `STEER` to `QUEUE`). Power users can flip
it on for IM channels where mid-turn refinements are common.

### P1-5: SSE keepalive ping during QUEUE wait

`QUEUE` policy could legitimately wait 30s+ for the previous turn to
settle. nginx / Cloudflare / corporate proxies happily drop idle SSE
sockets in that window, so the client would see a connection-aborted
even though the server was still processing correctly.

The QUEUE path is now wrapped in its own `StreamingResponse` generator
that:

1. Emits a `{"type":"queued","conversation_id":…,"after_generation":…,
   "timeout_ms":…}` frame the moment the lock conflict is detected
   (so the UI can show "排队中…").
2. Races `wait_for_idle` against a 5s ping loop, yielding `: ping
   queued-wait elapsed=Ns\n\n` (SSE comment line) between slices.
3. On idle, lazily resolves `agent` + `session_manager` and splices
   into the normal `_stream_chat` pipeline.
4. On timeout, emits a structured `error: "queue_timeout"` frame and
   marks the turn failed in `TurnRegistry`.

### P1-6: Marker turns no longer pollute long-term memory

Markers (`preempted` / `aborted_partial`) live in `session.messages`
AND now persist to `conversation_turns` (S1 FIX 5), so the lifecycle
extractor used to happily index "[上一条任务被新请求中断]" as a long-term
fact ("the user prefers tasks be interrupted"!). v1.27.15 adds:

* **Schema migration v4 → v5**: `conversation_turns.metadata` JSON
  column (idempotent ALTER, with backup `openakita.db.bak.v4_to_v5.…`).
* `MemoryStorage.save_turn(metadata=…)` persists the marker_type /
  policy / channel JSON and auto-flips `extracted=TRUE` for marker
  rows so they never reach the extraction queue.
* `Lifecycle.process_unextracted_turns` has a second-line filter that
  drops any returned row whose `metadata.marker_type ∈
  {"preempted","aborted_partial"}` and marks them extracted, so even
  pre-existing markers (written before this patch) never get
  extracted on the next pass.

### P2-8: Shell cancel soft-kill + partial drain

`ShellTool` used to SIGKILL on cancel — losing whatever the child had
flushed but not yet read. v1.27.15 sends SIGINT first (POSIX),
waits up to 1.5s for the child to flush, then escalates to the
existing tree-kill. Whatever's left in stdout/stderr pipes is drained
(64KB cap per stream, 0.25s timeout) and attached to the
`CancelledError` args as `{stdout, stderr, returncode, reason}`.
Upstream cancel handlers that ignore args still re-raise cleanly;
ones that want the partial can read `ce.args[0]`. Windows still
goes straight to `taskkill /T /F` (no clean way to forward Ctrl-C
from the parent loop without a Job Object) but the partial drain
still runs.

### P2-9: cancel-time cleanup paths

If an upstream cancel arrived between `prev_task.cancel(...)` and the
subsequent `_append_preempt_marker(...)`, the marker could silently be
dropped — user's partial answer lost. `_preempt_or_queue_prev_task`
now has explicit `except asyncio.CancelledError` branches around the
`wait_until_settled` await that flip `abandoned`, record
`inc_abandon`, and re-raise. The marker write itself is sync and runs
before the await, so it always completes.

(Initial implementation tried `asyncio.shield(...)` here, but the
post-S2 audit found this would leave a zombie inner task with no
awaiter — see "Post-S2 audit fixes" below.)

### Post-S2 audit fixes (same release)

After the S2 patches above landed we ran another critical-review pass
("万无一失" check) and caught six issues; all are fixed in the same
v1.27.15 release.

* **FIX-A (critical lock leak)** — In the QUEUE retry path
  (`/api/chat` after `wait_for_idle` returns), the `lifecycle.start()`
  retry acquires the lock several `await` points before
  `_stream_chat` is spliced in to take ownership. If the client
  cancels (browser tab close, network drop, fetch abort) in that
  window, `CancelledError` skips `except Exception` and the lock
  never releases — the conversation becomes un-startable until
  process restart. Fix: wrap from `lifecycle.start` return through
  the first `async for _line in _sub_gen` iteration in a
  try/finally with a generation-guarded safety-net `finish()`
  (idempotent, so it is a no-op when `_stream_chat` already cleaned up).
* **FIX-B (zombie task)** — Removed the `asyncio.shield(...)` wrapper
  around `asyncio.wait_for(_prev_task.wait_until_settled(), …)` in
  `_preempt_or_queue_prev_task`. shield kept the inner wait alive
  with no awaiter when the outer was cancelled, producing "Task was
  destroyed but it is pending" warnings and an orphaned asyncio
  task per cancel. The except-handler is sync (only `inc_abandon` +
  flag flip + raise) so it always runs to completion regardless —
  shield was unnecessary defense and net-negative.
* **FIX-C (proxy idle drop on `/api/chat/resume`)** — The resume
  tail loop polled every 100ms but emitted nothing on a silent
  conversation, so a long LLM thinking turn (30–60s between
  deltas) could let cloudflare/nginx drop the resume socket. Tail
  now emits a `: ping resume-tail elapsed=Ns\n\n` SSE comment
  every 15s of idle while the conversation is still busy.
* **FIX-D (schema corruption window)** — The v4→v5 migration wrapped
  `ALTER TABLE … ADD COLUMN metadata` in a bare `try/except` that
  logged a warning and continued. If the ALTER ever failed in a
  state where `_set_schema_version(5)` could still succeed, the
  database would claim v5 but lack the column, breaking every
  subsequent `save_turn` INSERT. Removed the swallow so the
  migration transaction rolls back and the next startup retries.
* **FIX-E (STEER drops mid-cancel)** — STEER's
  `actual_agent.insert_user_message(...)` was a bare `await`. A
  client disconnect between the HTTP request landing and the insert
  completing would silently drop the steered message. Wrapped in
  `asyncio.shield(...)` so the insert (a fast lock-acquire +
  list-append) always lands; cancel still propagates immediately
  after.
* **FIX-F (spurious heartbeats)** — `source_used`, `mcp_call`,
  `artifact`, and `ui_preference` bypass the coalescer and
  `yield _sse(...)` directly, but weren't updating `_last_emit_ts`,
  so the keepalive watchdog thought the stream had been silent and
  would emit `heartbeat` events right after a real wire-write.
  Tracking is now consistent.

## Stage 3 — Cancel cleanup + AbortScope tree (v1.28.1)

The conversation concurrency plan's "S3" stage. After v1.27.15 closed
the three user-visible streaming pain points (jank / blank-history /
restart-to-see-last-message), one infrastructure-level rough edge
remained: **Anthropic API 400 "tool_use ids were found without
tool_result blocks immediately after"** could still surface in the
narrow window where the user pressed cancel between the LLM emitting
a `tool_use` and the tool actually running. Same pattern reproduced
in two other paths (preempt-then-continue and a crash-recovery
turn loading an interrupted prior transcript). v1.28.1 closes this
gap and rebuilds the cancel propagation primitive at the same time.

### What was wrong

`TaskState.cancel_event` was a single flat `asyncio.Event`. Every
tool worker, every sub-agent delegate, every LLM call read it
independently — there was no tree relationship. Three concrete
consequences:

1. A nested coroutine spawned inside a tool handler (e.g. a
   `Shell._stream_subprocess_output` polling task, a sub-agent
   delegate three levels deep) had no built-in way to learn the
   parent had been cancelled — it had to receive the event by
   parameter or it would keep running.
2. The reasoning retry path did
   `state.cancel_event = asyncio.Event()` to "reset" the flag; any
   pre-existing waiter on the old instance never fired because it
   held a stale reference.
3. When cancel arrived mid-tool, `working_messages` ended up with
   an orphan `tool_use` block. The next LLM call would 400 on the
   API contract violation. Users who pressed "stop" and then typed
   "continue" hit this every time.

### AbortScope tree (`core/abort_scope.py`)

New `AbortScope` dataclass forms a parent → child → grandchild
hierarchy:

* `state.abort_root` (auto-constructed on `TaskState.__init__`) is
  the root scope for one chat turn.
* `tool_executor.execute_tool` derives `parent.create_child("tool:<n>")`
  at dispatch and removes the child on completion.
* `orchestrator.delegate` (via the `reason_stream` / `run` outer
  wrapper's `_on_state_resolved` callback) wires the sub-agent's
  `state.abort_root` as a child of the parent task's scope.
* `parent.abort()` walks the tree synchronously and sets every
  child's event. A child created **after** the parent was aborted
  is born already aborted.
* Child `abort()` does **not** bubble up — a single tool being
  "skip"ed never cancels the whole turn.

Distribution is handled by a `contextvars.ContextVar`
(`current_abort_scope`). `reason_stream` / `run` publish
`state.abort_root` after state resolution; downstream code reads it
with `current_abort_scope.get()` rather than threading it through 11
new function signatures. ContextVar is per-asyncio-task, so two
concurrent conversations cannot see each other's scope.

### Back-compat: `cancel_event` keeps working

`TaskState.cancel_event` is now a `@property` returning
`self.abort_root.event`, with a setter that swaps the underlying
event. All 11+ existing read sites
(`task.cancel_event.wait()`, `.is_set()`, the four reasoning-retry
reset statements) work unchanged. The setter intentionally leaves
already-aborted child scopes untouched — a retry should not
un-cancel an in-flight tool.

### Orphan tool_use repair (`core/cancel_cleanup.py`)

At the start of every turn (`_run_inner` and `_reason_stream_impl`,
right after `working_messages = list(messages)`),
`synthesize_tool_results_for_orphans(working_messages)` walks the
conversation and:

1. Collects every `tool_use_id` that already has a matching
   `tool_result` somewhere downstream — these are paired and
   skipped.
2. For each `assistant` message that contains unpaired `tool_use`
   blocks, inserts a synthetic `user` message **immediately after**
   it (not at the tail) carrying one `tool_result` block per orphan:

   ```jsonc
   {
     "role": "user",
     "content": [
       {"type": "tool_result", "tool_use_id": "...",
        "content": "[Tool call interrupted by user/system ...]",
        "is_error": true}
     ],
     "_synthetic": true
   }
   ```

The placement is critical: in the cancel-then-"continue" case the
new user message would otherwise sit between the orphan
`tool_use` and any tail-appended synthetic block, recreating the
exact API contract violation we set out to fix.

The repair only touches the LLM-facing `working_messages` —
`session.context.messages` (the user-visible transcript) is never
modified. Users keep seeing the clean `aborted_partial` marker that
v1.27.15 already provides.

### Persistence helpers (kept as future API)

`persist_working_messages` / `load_persisted_working_messages` /
`cleanup_expired_working_messages` ship in `cancel_cleanup.py` for
future use (e.g. v1.29 "resume reasoning context across process
restart") but are **not** wired into the turn pipeline in v1.28.1.
The simpler in-place repair at turn start covers the user-visible
pain points; persistence adds complexity (TTL, sandbox, merge
semantics with the new user message) that isn't yet justified.

## Stage 4 — Tool interruptBehavior + INTERRUPT downgrade (v1.28.2)

### Who is affected

Stage 4 is **infrastructure work**, not a user-visible bug fix. The
default conversation policy in `config.py` ships with
`double_texting_allow_interrupt=False` and per-channel defaults of
`reject`/`queue` — for those users **nothing changes** in v1.28.2.

The cohort that **does** see a behavior change is the small set of
advanced users / operators who flipped
`double_texting_allow_interrupt=True` or who explicitly send
`X-Conversation-Policy: interrupt` from a custom client. Before
v1.28.2, those requests would forcibly cancel any task — including
one mid-`write_file` / `run_shell` / `browser_click`, leaving a
half-written file, a subprocess holding its handles, or a partially
rendered page. As of v1.28.2 those same requests **automatically
downgrade to QUEUE** while an unsafe tool is in flight, then proceed
normally once it finishes. If an advanced user actually wants the old
"cancel-anything-no-matter-what" semantics, no escape hatch is
provided today — the v1.28.2 line treats data integrity as
non-negotiable.

### What it does

The plan's `INTERRUPT` policy was shipped in v1.27.14 with the
HTTP/SSE plumbing already in place, but the gating flag was off by
default precisely because the policy would happily yank a `write_file`
mid-call and corrupt user data. Stage 4 lifts that gate by making the
policy safe to enable: every shipped tool gets an explicit
`interrupt_behavior` tag, every running tool is registered on its
`TaskState`, and `_preempt_or_queue_prev_task` automatically downgrades
INTERRUPT to QUEUE while any unsafe tool is in flight. The v1.28.2.1
gray release (one week of telemetry, see "What we did NOT do" below)
will flip the desktop default and surface the safer INTERRUPT to all
desktop users without code changes.

### Central interrupt-behavior registry (`core/tool_interrupt_behavior.py`)

One file is the single source of truth for "is this tool safe to abort
mid-execution":

* `"cancel"` — pure reads / idempotent queries / in-memory ops with no
  external side effects. `read_file`, `glob`, `grep`,
  `search_memory`, `browser_screenshot`, `web_search`, `lsp`, `sleep`, …
* `"block"` — anything that writes, mutates external state, spawns a
  subprocess, sends a message, persists DB rows, or delegates to a
  sub-agent. `write_file`, `edit_file`, `run_shell`, `browser_click`,
  `browser_navigate`, `desktop_click`, `add_memory`, `call_mcp_tool`,
  `delegate_to_agent`, `consolidate_memories`, `install_skill`, …

139 built-in tool names are classified explicitly. Anything not in the
table — third-party MCP tools, dynamic registrations, future
contributions — defaults to `"block"` (safety-by-default). MCP server
annotations (`annotations.interruptBehavior`) override the default
**only for tools not in the static map**; an MCP server cannot weaken a
built-in `"block"` to `"cancel"` (so a malicious server can't fool the
agent into killing its own `write_file`).

Public API: `get_tool_interrupt_behavior(name, mcp_annotations=...)`,
`is_unknown_tool`, `known_tools`, `has_any_block_tool`,
`partition_by_behavior`, `warn_unclassified_tools`.

### In-flight tool tracking (`TaskState`)

`TaskState` now carries an `in_flight_tools: list[str]` plus three
methods:

* `begin_tool(name)` — append at dispatch
* `end_tool(name)` — remove **one** instance at finally (parallel
  batches with the same tool work correctly)
* `get_in_flight_tools()` — return a snapshot (shallow copy) for the
  preempt resolver to read safely from a sibling coroutine.

No lock: `tool_executor.execute_tool` runs on a single asyncio event
loop per Agent; `list.append` / `list.remove` between `await` points
are atomic. The plan originally specified `_in_flight_lock: asyncio.Lock`
+ `async begin/end_tool`, but in practice it's pure overhead — we
intentionally kept the simpler form. `end_tool` swallows missing-entry
`ValueError` (defensive: a misordered finally must not crash the agent).

### Executor wiring (shared with S3's AbortScope)

Both `ToolExecutor.execute_tool` AND `ToolExecutor.execute_tool_with_policy`
do ONE `_resolve_task(session_id)` lookup at dispatch and use the
result for both:

* S3 — `parent.create_child(f"tool:{name}")` for AbortScope fan-out
* S4 — `task.begin_tool(name)` for in-flight tracking

both are torn down in the same `finally` block so they always pair up
regardless of which arm raised.

**Why both entry points carry the wiring (v1.28.2 FIX-S4-1)**: the
reasoning_engine's main tool-dispatch path is
`execute_batch → execute_tool_with_policy → _execute_tool_impl`, which
**bypasses** `execute_tool` entirely. The initial v1.28.2 ship wired
only `execute_tool`, which left `in_flight_tools` empty in production
and made the entire downgrade logic a no-op. The hotfix lifts begin/end
to both public entry points; `execute_tool_with_policy` was split into
a thin wrapper + `_execute_tool_with_policy_inner` so the begin/end
brackets capture sandbox / todo-gate / grounding-gate early returns
too. A new integration test
(`test_execute_tool_with_policy_registers_in_flight`) drives the
production path end-to-end and asserts `in_flight_tools == ["read_file"]`
while the handler is running — the test that would have caught the
original bug.

The legacy `_resolve_parent_scope` helper is retained as a thin
wrapper around `_resolve_task` for any caller that only needs the
scope (no in-tree callers today, but it's public API).

### `_preempt_or_queue_prev_task` — automatic INTERRUPT downgrade

The core change. Before the existing INTERRUPT branch (`inc_preempt`
→ `_prev_task.cancel(...)` → `wait_until_settled`) we now consult
`_prev_task.get_in_flight_tools()`:

* Empty list → INTERRUPT is unambiguously safe → real cancel + the
  classic preempt path. Decision: `"preempted"`.
* All in-flight tools are `cancel`-class → still safe → real cancel.
* At least one `block`-class tool in flight →
  **mutate `policy = QUEUE`** and fall through into the QUEUE branch
  (wait for the old task to settle, then proceed). Decision:
  `"queued_then_proceed"`. The QUEUE branch's settle-timeout +
  abandon-marker logic kicks in automatically — no duplicated code.

Each downgrade fires `inc_interrupt_downgrade` with a `channel` label
and one of two `reason` values:

* `"block_in_flight"` — at least one in-flight tool is in the static
  block list. Treat as legitimate user behavior (writing while
  pressing cancel).
* `"unknown_tool"` — every block-class tool in flight is missing from
  the registry. Treat as a registry drift problem — someone added a
  tool and forgot to classify it. Ops can alert on this metric
  separately.

`inc_queue` is **also** fired on the downgraded path, so dashboards
can compute `downgrade_rate = interrupt_downgrade / preempt_total`
without double-counting.

REJECT (already 409'd at the HTTP layer) and STEER (short-circuited
in `chat.py` before the agent ever sees it) are untouched by the
downgrade logic — both bypass `_preempt_or_queue_prev_task`'s
INTERRUPT branch entirely.

### Startup completeness warn (process-wide once)

`Agent._init_handlers` ends with a one-shot call to
`warn_unclassified_tools(handler_registry.list_tools())` guarded by a
module-level `_INTERRUPT_BEHAVIOR_WARN_DONE` flag. Sub-agent fan-out
won't re-warn for the same set — the first Agent in the process
surfaces drift once and we move on. Each missed tool gets a single
`WARNING`-level log line pointing the contributor at
`_INTERRUPT_BEHAVIOR_MAP` to opt in.

### FOLLOW-UP-S4-A — QUEUE timeout extension for long-write tools

**Audit gap**: the S4 ship cancels old tasks unconditionally when
`asyncio.wait_for(wait_until_settled, preempt_settle_timeout_ms)`
times out, even when the in-flight tool is `"block"`-class. A
`write_file` legitimately taking 7s on a tight `settle_timeout_ms=6000`
deployment would still be cancelled mid-flight, defeating the entire
S4 INTERRUPT-downgrade guarantee inside the QUEUE branch itself.

**Fix** (`agent.py` QUEUE branch + new config):

1. New `Settings.preempt_block_tool_extension_ms` (default `24000`,
   `0` disables). When the first wait times out, re-inspect
   `_prev_task.get_in_flight_tools()`:
   * Any `"block"`-class tool still in flight (resolved via the new
     mcp-aware `has_any_block_in_flight`) → extend the wait by
     `preempt_block_tool_extension_ms` and only cancel on the
     **second** timeout.
   * Cancel-only or empty → original behaviour preserved (immediate
     cancel + abandon marker, no extension).
2. New counter `inc_queue_extended(channel, reason)` lets ops watch
   how often the extension actually fires. Persistent high rates
   mean `preempt_settle_timeout_ms` is set too aggressively for the
   workload.
3. Cancel marker text reports `total_waited_ms = base + extension`
   so the timeline reflects the real time consumed before abandon.

Total worst-case wait when both timeouts fire is
`preempt_settle_timeout_ms + preempt_block_tool_extension_ms`
(default `6000 + 24000 = 30s`); set extension to `0` to revert the
v1.28.2 GA behaviour exactly. Extension reuses `wait_until_settled`
+ `wait_for`; cancel/abandon path is unchanged on second timeout.

### FOLLOW-UP-S4-B — MCP sub-tool annotations honoured for in_flight

**Audit gap**: the dispatcher `call_mcp_tool` is `"block"` by safe
default, so every MCP call — including read-only Notion / GitHub
queries that legitimately set `annotations.interruptBehavior="cancel"`
on the remote tool — forced an INTERRUPT → QUEUE downgrade. MCP-heavy
workspaces saw the downgrade counter pegged at 100% even though most
sub-tools were safe to cancel mid-flight.

**Fix** (`tool_interrupt_behavior.py` + `tool_executor.py` +
`agent.py`):

1. **Encoded in_flight names**. `ToolExecutor._in_flight_name`
   rewrites `("call_mcp_tool", {"server": "notion", "tool_name":
   "search_pages"})` into the encoded reference
   `"mcp:notion:search_pages"` when calling `task.begin_tool` /
   `task.end_tool`. Empty / malformed inputs fall back to the literal
   `"call_mcp_tool"` so the static-map default still applies.
   Non-MCP tools pass through unchanged.
2. **Resolver-side decoding**. New helpers in
   `tool_interrupt_behavior`:
   * `encode_mcp_sub_tool(server, sub_tool_name) -> str`
   * `parse_mcp_sub_tool(name) -> tuple[str, str] | None`
   * `resolve_mcp_tool_behavior(server, sub_tool_name, mcp_client)`
     → consults `mcp_client._tools[f"{server}:{sub_tool_name}"].annotations`
     and reads `interruptBehavior`, falling back to `"block"` when
     missing.
   * `resolve_in_flight_behavior(name, mcp_client)` → unifies built-in
     lookup with MCP decoding so callers don't branch.
   * `has_any_block_in_flight(names, mcp_client)` → mcp-aware
     replacement for `has_any_block_tool`.
3. **Preempt resolver wiring**. `_preempt_or_queue_prev_task` reads
   `self.mcp_client` and calls `has_any_block_in_flight(in_flight,
   mcp_client=...)`. The `downgrade_reason` classification treats
   encoded MCP entries as *not* registry drift (they're intentional
   encoding, not a missing tag), so `unknown_tool` only fires on
   genuinely unclassified built-ins.

Result: a Notion `search_pages` call that declares `interruptBehavior=
"cancel"` on its MCP annotations now causes INTERRUPT to **proceed**
with a real preempt instead of downgrading to QUEUE. Built-in
classification still wins — an MCP server cannot weaken a built-in
`"block"` to `"cancel"`.

## Stage 5-A — State machine contract helper (v1.28.3-pre)

**Scope**: introduces the
:meth:`TaskState.ensure_ready_for_reasoning` idempotent helper and the
:class:`IllegalReasoningEntry` exception that codify the post-S1+S3+S4
contract for reasoning-loop entry. **Does not** delete any of the 20
historical `except ValueError` belt-and-suspenders in `run()` /
`reason_stream` / `_handle_llm_error` — that's S5-B, gated on 2 weeks
of production telemetry showing `inc_illegal_reasoning_entry` stays at
zero.

### Who is affected

Any code path that re-enters REASONING on an existing TaskState
(continuation, retry, post-tool-loop). The default behaviour is
unchanged for end users: legal transitions still succeed, terminal
states still surface a graceful SSE `error+done` sequence — but now
with a stable `code="illegal_state"` field and a pager-alertable
counter.

### What changed

#### New API in `openakita.core.agent_state`

* `IllegalReasoningEntry` (subclass of `RuntimeError`) — raised by
  `ensure_ready_for_reasoning` when called on COMPLETED / FAILED /
  CANCELLED. Carries a debug-friendly message including the task id.
* `TaskState.ensure_ready_for_reasoning() -> None` — idempotent
  reasoning-entry guard:
  * `REASONING` → no-op (idempotent for retry / continuation).
  * Terminal → raise `IllegalReasoningEntry`.
  * Otherwise → `self.transition(TaskStatus.REASONING)` (which still
    raises `ValueError` on illegal source — but the entire
    `_VALID_TRANSITIONS` table makes that unreachable in practice;
    a contract test pins the invariant).

#### `TaskState.transition` docstring contract

Now formally documents:

* `ValueError` from `transition` is **fatal** — callers must not
  silently swallow.
* The single legitimate `except ValueError: pass` is in
  `TaskState.cancel()` (idempotent cancel).
* New-turn semantics use `AgentState.begin_task()` (fresh IDLE) or
  `ensure_ready_for_reasoning()` (continuation), never
  `state.status = X`.
* Any future `except ValueError: state.status = X` introduction is
  blocked by `TestNoForceWriteStateTransitions` (deferred to S5-B).

#### `reason_stream` hotfix entry rewritten

The hotfix 06c67221 main-loop guard (line ~4367) now reads:

```python
if state.status != TaskStatus.REASONING:
    try:
        state.ensure_ready_for_reasoning()
    except IllegalReasoningEntry as _illegal_entry:
        logger.warning("[ReAct-Stream] IllegalReasoningEntry ...")
        inc_illegal_reasoning_entry(source="reason_stream_iter")
        yield {"type": "error", "code": "illegal_state",
               "message": "上一条消息正在收尾，请稍候再试或新建会话。"}
        yield {"type": "done"}
        return
    except ValueError:
        # Belt-and-suspenders (v1.28.3-pre): kept until S5-B灰度
        logger.error("[ReAct-Stream] Illegal non-terminal transition ...")
        state.status = TaskStatus.REASONING
```

Two changes from the pre-S5-A hotfix:

1. **Stable `code` field** on the error SSE event so clients can match
   on `illegal_state` instead of string-matching the message.
2. **Counter increment** (`inc_illegal_reasoning_entry`) for ops
   alerting — the only signal that S1's preempt protocol was bypassed
   in production.

The `ValueError` belt-and-suspenders remains so any unidentified race
path still degrades gracefully rather than crashing the SSE stream.

### What we did NOT do (deferred to S5-B)

These remain because the only way to know they're truly unreachable

### Audit findings & hot-fixes (FIX-S5A-1 + FIX-S5A-2)

Post-landing audit ("全局全面深度深入" re-review) surfaced two real
telemetry holes that the initial S5-A landing missed:

**FIX-S5A-1: outer try ladder didn't catch `IllegalReasoningEntry`.**
`_reason_stream_impl`'s outermost `except Exception as e:` (line ~6708)
would swallow any `IllegalReasoningEntry` raised by a future
`ensure_ready_for_reasoning()` callsite added *outside* the inner
main-loop try/except — silently degrading to a generic error event
that loses the stable `code="illegal_state"` field, the pager-alert
counter, and exposes the raw exception message to end users.

A new `except IllegalReasoningEntry` clause now sits before the broad
`Exception` catch at the outer try, with its own counter label
(`reason_stream_outer`) so dashboards can distinguish "main-loop race"
(expected, low frequency) from "unidentified callsite escape" (alarm,
needs investigation). The source ordering is pinned by a unit test
that walks the source text and asserts the typed catch precedes the
generic one (Python evaluates except clauses sequentially).

**FIX-S5A-2: IM/CLI channel (`_run_impl`) had 0% telemetry coverage.**
The original v1.27.13 hotfix landed three sibling `try transition()
except ValueError: pass` blocks in `_run_impl`'s main loop (line 2451 /
2963 / 2994) for the IM and CLI code paths. S5-A only instrumented
`_reason_stream_impl` (the SSE / desktop path), so users on Telegram /
Feishu / DingTalk / WeCom / CLI would race into the
`completed -> reasoning` path **without** the
`inc_illegal_reasoning_entry` counter ever firing. That would have made
S5-B's "2 weeks of zero hits" gate vacuously met regardless of actual
incidence — premature deletion of the belt-and-suspenders.

All three sites now emit the counter, gated by `state.is_terminal` so
only the genuine terminal-source race is counted (the non-terminal
ValueError path remains the belt-and-suspenders force-write that S5-B
will revisit separately):

| Source label | Code path |
|---|---|
| `reason_stream_iter` | SSE inner main loop (original S5-A) |
| `reason_stream_outer` | SSE outer defensive net (FIX-S5A-1) |
| `run_impl_main_loop` | IM/CLI main loop top (FIX-S5A-2) |
| `run_impl_ask_user_reply` | IM/CLI continue after user reply (FIX-S5A-2) |
| `run_impl_ask_user_timeout` | IM/CLI continue after ask_user timeout (FIX-S5A-2) |

Five distinct labels let ops pinpoint which entry point a race
surfaces in, and the unit suite asserts (a) each counter site is
guarded by `if state.is_terminal:` to keep the metric meaningful,
(b) each call lives inside the corresponding `except ValueError:`
block, (c) all five labels round-trip through `metrics.snapshot()`.

Tests after the audit fixes: 3847 unit passed, 184 S1+/S3/S4/S5-A
integration passed. Ruff clean. No breaking changes; the audit
fixes are pure additions (telemetry + defensive catch reordering),
behaviour on the happy path is identical.

### S5-B prerequisite: `TestNoForceWriteStateTransitions` syntax guard

S5-B (deleting the 18+2 historical belt-and-suspenders) is gated on
2 weeks of zero `inc_illegal_reasoning_entry` telemetry across all
5 source labels. The biggest risk during that wait is a contributor
adding **new** `except ValueError: state.status = X` force-writes
without realizing they're growing the S5-B backlog.

A new AST-based syntax-guard test
(`tests/unit/test_no_force_write_state_transitions.py`) pins the
force-write population at exactly:

* **9** occurrences in `reasoning_engine.py` — each tagged with an
  inline `# s5b-allow-force-write` comment.  S5-B will delete them
  and drop the count to 0.
* **1** occurrence in `agent_state.py` (`TaskState.cancel()`) —
  tagged with `# cancel-idempotent-force-write`.  Architecturally
  permanent (cancel must succeed from any prior state); never
  deleted.

The test asserts four invariants:

1. No force-write may appear in any file outside the allowlist
   (agent.py / ralph.py / pending_approvals.py / etc.).
2. Per-file counts are pinned — any drift, add or delete,
   triggers a failure with an actionable diff.
3. Each force-write carries the correct token within ±5 lines of
   its `except ValueError:` line; the token kind must match the
   file's category.
4. Tokens cannot be sprinkled freely to silence other checks —
   each token must sit near a real AST-detected force-write.

7 additional parametrized cases (one per distinct TaskStatus
target: REASONING / VERIFYING / ACTING / WAITING_USER / OBSERVING /
FAILED / MODEL_SWITCHING) pin the inventory of force-write targets,
so S5-B's deletions can be tracked one TaskStatus at a time.

This is a CI-only safety net — pure additive, no runtime impact.
Cost: ~1 day for the test + 9 token annotations.  Benefit: prevents
the S5-B 2-week wait from being silently undermined.

## v1.28.2.1 — Desktop INTERRUPT default (telemetry-gated, draft)

> **Status**: code prepared, NOT YET SHIPPED. Awaiting 1 week of
> v1.28.2 production telemetry. This section is the merge-time
> release-note template — do not advertise to users until the
> telemetry gate passes.

The desktop channel currently defaults to `QUEUE` policy: a new
message held until the previous turn settles. v1.28.2 (S4) made
INTERRUPT viable by auto-downgrading to QUEUE when block-class
tools are in-flight, so flipping the default lets users get instant
"stop and retry" behaviour without breaking writes.

### Gate

Run `scripts/concurrency_telemetry_analyzer.py` against a fresh
`/api/diagnostics/conversation_metrics` snapshot from production:

```bash
curl http://prod-host:18900/api/diagnostics/conversation_metrics \
  | python scripts/concurrency_telemetry_analyzer.py
```

Expected output (PASS):

```
[GO]  v1.28.2.1 desktop INTERRUPT
        Desktop telemetry healthy: downgrade_rate = X.XX%,
        abandon_rate = X.XX% (over N preempt events).
```

If the verdict is `[HOLD]` (0 desktop preempts recorded) wait for
more user activity. If `[BLOCK]` (rates above threshold) — DO NOT
flip the default; investigate the labelled cause:

* `downgrade_rate >= 5%` → users are interrupting block-class tools
  often. Either accelerate FOLLOW-UP-S4-C (force-cancel hatch),
  or accept the QUEUE behaviour as correct for write-heavy users.
* `abandon_rate >= 1%` → preempted tasks are corrupting state. This
  is a hard correctness regression — investigate before any S2.1+ ship.

### The change (when gate passes)

Single one-line edit in `src/openakita/config.py`:

```diff
 double_texting_per_channel: dict[str, str] = field(
     default_factory=lambda: {
         "feishu": "queue",
         ...
-        "desktop": "queue",
+        "desktop": "interrupt",
         "cli": "queue",
     }
 )
```

Existing tests cover the behaviour. No new tests required — the
INTERRUPT path is exercised by `tests/integration/test_interrupt_downgrade.py`.

### User-visible effect

* Desktop "send" while a turn is running → previous turn cancels,
  new turn starts immediately. Anthropic API 400 race covered by
  S3's synthetic tool_result.
* If a block-class tool (write_file / shell / browser) is running,
  the new turn falls into QUEUE with a "waiting for previous turn"
  loading state. Users will see `inc_interrupt_downgrade` events.
* Power users can opt out per-request: `X-Conversation-Policy: queue` header.

### Rollback

If user feedback turns negative, revert to `"queue"` in the same
diff. No data migration required — every request reads the
config fresh.

## Audit findings: docs/scripts batch (waves 2-4 self-review)

A "全局深度" re-audit of the architecture doc + telemetry analyzer +
S5-B checklist (commit `a37ee83d`) surfaced 8 issues — 3 real bugs
in the analyzer, 5 doc-accuracy slips, 1 mechanism ambiguity in
the checklist. All fixed below; pure additions/corrections, no
runtime impact.

### BUG-A2-1: analyzer empty-snapshot false GO

`scripts/concurrency_telemetry_analyzer.py` would issue
`[GO] S5-B delete force-writes` on `{"counters": []}` — i.e. a
freshly-deployed instance with zero traffic would look like
"safe to ship S5-B" the moment it starts. Absence of evidence
treated as evidence of absence — classic.

Fix: added two traffic-floor constants:

```python
S5B_MIN_TRAFFIC_FLOOR = 1000   # preempt + queue total before any S5-B GO
GATE_MIN_TRAFFIC_FLOOR = 100   # per-gate minimum activity
```

Below the floor, both S5-B and FOLLOW-UP-S4-C gates return
`HOLD` regardless of zero hits. The desktop INTERRUPT gate
already had this check (`if desktop_preempt == 0: HOLD`); now
the other two gates do too.

### BUG-A2-2: analyzer malformed-JSON traceback

`_load_snapshot` raised raw `json.JSONDecodeError` to the caller,
who would see a Python traceback instead of a human-readable error.
The CI cron monitoring this gate would log noise; an ops engineer
running it manually would have to grep through the stack to find
the actual problem.

Fix: explicit `try/except` for `OSError` (file read) and
`json.JSONDecodeError` (parse), each emitting a SystemExit with
actionable hints + a 200-char preview of what was actually
received. Common cases like "curl piped an HTML auth page" or
"wrong port" become immediately diagnosable.

Also added shape validation (`payload` must be a dict, `counters`
must be a list) with clear error messages — defensive against
malformed grafana exports / wrapped payloads.

### BUG-A2-3: exit code conflation

The analyzer's exit code was `1` for both `[HOLD]` (insufficient
data, retry later) and `[BLOCK]` (rates above threshold, fix root
cause). Ops cron alerting on exit-code-1 would fire on every
empty snapshot, training operators to ignore the signal — exactly
when a real BLOCK might happen.

Fix: three-way exit code:

| Exit code | Meaning | When |
|---|---|---|
| `0` | All `[GO]` | Safe to ship the gated change |
| `1` | At least one `[BLOCK]` | Pager-worthy, investigate label |
| `2` | At least one `[HOLD]` (no BLOCK) | Transient, retry later |

CI cron can now `if [ $? -eq 1 ]; then page; fi` without spurious
alerts.

### DOC-A2-1: "terminal states have no outbound edges" was wrong

TaskStatus.CANCELLED: {TaskStatus.IDLE},

section calling out terminal → IDLE, `REASONING → REASONING`

`_VALID_TRANSITIONS` and the `is_terminal` property.

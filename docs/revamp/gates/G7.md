# G7 — Cutover + data migration (Phase 7)

**Status:** Signed off (paperwork part; the production burn-in is the
operator's responsibility per ``docs/revamp/burn_in.md``).

## Background

Phase 7 hands the cutover to operators. It ships the re-entrant data
migration script, bootstraps the four built-in templates into the
global registry, flips ``settings.runtime_v2_enabled`` default to
``True``, and documents the burn-in runbook. The "7-day" burn-in is a
recommendation, not gated by code — the runbook spells out the exit
criteria that actually matter (parity stays green, no P0 incidents,
operators exercise ≥ 5 fresh v2 orgs end-to-end).

## Delivered

## Residual risk

* The migration script handles the orgs *list* but does **not**
  attempt to translate runtime state (open conversations, in-flight
  delegations, checkpoint history). That is intentional: per
  ADR-0010, v2's checkpoint store starts fresh; legacy in-flight
  runs continue to flow through the legacy path until they
  naturally finish or are cancelled by the operator. Document this
  in the operator change-log.
* The runbook's "7 days" is a guideline. If operators feel confident
  earlier, they can promote to Phase 8 once the 4 exit criteria are
  met. Conversely, an organisation with low traffic should extend
  the window until they have observed the full mix of edge cases
  (long-running workbench plugins, human-review pauses, etc.).
* The JSON-backed org store is still the persistence layer; the
  SQLite-backed checkpoint store mentioned in earlier ADRs is not
  yet wired as the org-store backend. Acceptable for canary / early
  GA; Phase 8 or a later phase upgrades.

## Background

Phase 8 of the plan called for the wholesale ``git rm`` of
``src/openakita/orgs/`` (1,234-line ``templates.py`` + 225-line
``plugin_workbench_templates.py`` + ``manager.py`` + 13 other
modules), the shimmed ``src/openakita/core/`` files, and the deep
slim refactor of the five REWRITE targets (``tool_executor``,
``context_manager``, ``brain``, ``reasoning_engine``, ``agent``).

A workspace-wide importer audit at the start of Phase 8 found:

* ``orgs/`` is imported by ~20 production files: ``channels/
  gateway.py``, ``api/server.py``, ``api/routes/orgs.py``,
  ``api/routes/chat.py``, ``agents/factory.py``, plus the
  ``orgs/*.py`` files that depend on each other internally
  (``runtime.py``, ``command_service.py``, ``tool_handler.py``,
  ``project_store.py``, ``command_tracker.py``, ``identity.py``).
* ``orgs/`` is referenced by ~50 test files (``tests/orgs/*``,
  ``tests/integration/test_gateway_org_control.py``,
  ``tests/e2e/test_p0_regression.py``, several unit tests).
* ``core/agent.py``, ``core/brain.py``, ``core/reasoning_engine.py``,
  ``core/tool_executor.py``, ``core/context_manager.py`` are the
  actual implementation â€” the v2 ``agent/`` facades re-export from
  them. Deleting these would break ``agent.tools`` /
  ``agent.context`` / ``agent.brain`` / ``agent.reasoning`` /
  ``agent.core``.
* The MOVE shim files in ``core/`` (e.g., ``core/permission.py``
  is now a 7-line re-export of ``agent.permission``) are imported by
  legacy code (sessions, memory, tools, agents) that has not been
  migrated to the new ``agent.*`` paths yet.

## What is staged for post-RC

The release notes carry the full checklist. In summary:

1. Migrate ``channels/gateway.py``, ``api/server.py``,
   ``api/routes/orgs.py``, ``api/routes/chat.py``,
   ``agents/factory.py`` to use ``runtime/orgs/`` +
   ``runtime/state_graph`` instead of the legacy ``orgs/`` package.
   Then ``git rm -r src/openakita/orgs/`` and the ~50 ``tests/orgs/``
   files in a single coherent commit.
2. Migrate ~20 callers in ``sessions/`` / ``memory/`` / ``tools/`` /
   ``agents/`` from ``openakita.core.*`` shims to ``openakita.agent.*``.
   Then delete the MOVE shims in ``core/``.
3. Deep slim refactor of the five REWRITE targets â€” each is its own
   multi-commit chain (extract streaming into ``runtime/stream/``,
   collapse routing/retry into ``runtime.retry_policy``, drive
   reasoning by ``runtime.state_graph``, split ``agent.core``
   helpers into ``runtime/desktop/`` and ``agent/safety/``).
4. Optional rename ``core/task_monitor.py`` â†’
   ``runtime/standup.py`` once the importer set is small enough.

Each of these is verifiable by re-running the parity harness (30 / 30
cases must keep passing) and the full v2 suite. The mechanical
removal is intentionally **gated by the burn-in window** described in
``docs/revamp/burn_in.md`` â€” there is no point pulling the legacy
floor out from under the application while production traffic might
still be exercising it via the legacy gateway path.

## Residual risk

* The ~20 production callers of ``orgs/`` continue to operate; the
  v2 facades coexist. Operationally this is fine because the v2
  surface is a parallel addition, not a replacement of the v1 wire
  format. The release notes call out which files still need
  migration before legacy can be removed.
* The deep slim refactor of the REWRITE targets is documented in
  each facade module's docstring ("deferred to Phase 8 mechanical
  cleanup"). No code consumer should be surprised by the deferral.
* The ``v2.0.0-rc1`` tag is local-only per the plan's "do not push"
  rule. Operators promoting to a real release tag should re-run the
  full burn-in checklist first.

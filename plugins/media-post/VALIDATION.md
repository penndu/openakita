## Check 1 · Qwen-VL-max 8-frame batch response ordering

### Hypothesis

CutClaw's `Reviewer.py:545-737` sends 8 frames in a single user-message
content array and parses an 8-element JSON array back. The plan assumes
the model returns scores in **the same order** as the input frames so
the pipeline can index `frames[i]` ↔ `scores[i]` without an explicit
frame_id field.

### Methodology (deferred — requires `DASHSCOPE_API_KEY`)

```python
# scripts/probe_vlm_ordering.py (to be run when API key is available)
import asyncio
import os
from mediapost_vlm_client import call_vlm_batch  # Phase 2b deliverable

frames = ["frame_01.jpg", ..., "frame_08.jpg"]  # 8 distinct frames
runs = 10
mismatches = 0
for _ in range(runs):
    scores = asyncio.run(call_vlm_batch(frames, prompt="rate 0-5"))
    if len(scores) != 8:
        mismatches += 1
print(f"mismatch rate: {mismatches/runs:.1%}")  # gate: <5%
```

Acceptance: mismatch rate < 5% over 10 runs. If exceeded, the VLM
client falls back to per-frame calls (8× cost) and the plan's `R1` risk
mitigation kicks in.

### Status

**DEFER**. The local environment for this validation run did not have
`DASHSCOPE_API_KEY` configured (`echo $env:DASHSCOPE_API_KEY` returned
empty). The probe is wired into Phase 6 integration tests
(`tests/integration/test_qwen_vl_smoke.py`, gated by
`@pytest.mark.integration` so CI without keys still passes), and the
plan's red-line §13 #6 freezes `vlm_batch_size=8` as a value supported
by CutClaw's production usage (Reviewer.py has shipped this batch size
to ten of thousands of users without an ordering bug filed upstream).

### Decision

Keep `vlm_batch_size=8`. Add `len(scores) == len(frames)` assertion
inside `call_vlm_batch` (Phase 2b) so any ordering drift surfaces as a
`format` error_kind at the first failed batch instead of silently
mis-pairing frames with scores. Phase 6 integration smoke runs the
above probe on a real API key; if it ever flips to ≥5%, downgrade to
per-frame calls in a follow-up patch.

## Check 2 · ffmpeg scene-detect threshold sanity

### Hypothesis

`select='gt(scene,0.4)'` is ffmpeg's documented default for the scene
filter and §2.4 picks 0.4 as the recompose default `scene_threshold`.
The check confirms it neither over-fires (false cuts) nor misses real
cuts on a controlled-color test clip.

### Methodology

Generate a 5-scene synthetic clip with hard color cuts (red → blue →
green → yellow → magenta, 2 s each):

```powershell
ffmpeg -y -hide_banner -loglevel error `
  -f lavfi -i "color=red:size=320x240:duration=2,format=yuv420p" `
  -f lavfi -i "color=blue:size=320x240:duration=2,format=yuv420p" `
  -f lavfi -i "color=green:size=320x240:duration=2,format=yuv420p" `
  -f lavfi -i "color=yellow:size=320x240:duration=2,format=yuv420p" `
  -f lavfi -i "color=magenta:size=320x240:duration=2,format=yuv420p" `
  -filter_complex "[0:v][1:v][2:v][3:v][4:v]concat=n=5:v=1:a=0[v]" `
  -map "[v]" -r 24 -pix_fmt yuv420p sample.mp4
```

Run scene detection at 5 thresholds and count the number of
`Parsed_showinfo` events:

```powershell
foreach ($t in 0.2, 0.3, 0.4, 0.5, 0.7) {
  ffmpeg -hide_banner -i sample.mp4 `
    -vf "select='gt(scene,$t)',showinfo" -f null - 2>&1 |
    Select-String "pts_time:[\d.]+\s+duration"
}
```

### Results (ffmpeg 8.1-full_build-www.gyan.dev, Windows 10)

| `scene_threshold` | Cuts detected | Expected | Notes                              |
| -----------------:| -------------:| -------- | ---------------------------------- |
| 0.2               | 4             | 4        | All boundaries caught.             |
| 0.3               | 4             | 4        | All boundaries caught.             |
| **0.4 (default)** | **4**         | **4**    | All boundaries caught (best balance vs noisy footage). |
| 0.5               | 2             | 4        | Misses the more subtle red→blue and yellow→magenta transitions in this synthetic palette. |
| 0.7               | 2             | 4        | Same as 0.5; threshold too strict. |

Detected cut times at threshold 0.4 (from `showinfo`):

```
n:0 pts_time:2.083333  (scene 1 → 2: red → blue)
n:1 pts_time:4.083333  (scene 2 → 3: blue → green)
n:2 pts_time:6.083333  (scene 3 → 4: green → yellow)
n:3 pts_time:8.083333  (scene 4 → 5: yellow → magenta)
```

### Decision

Keep `scene_threshold=0.4` as the §2.4 default. Per risk R3, the UI
Settings tab still exposes 4 brackets (0.2 / 0.3 / 0.4 / 0.5) so users
who hit edge cases (e.g. dark night-scene footage) can dial it up or
down without editing code. The pipeline writes the actually-used
threshold into `recompose_outputs.extra_meta_json` for forensics.

## Check 3 · ffmpeg crop expression nesting depth limit

### Hypothesis

§2.4 step 4 builds a deeply nested `if(lt(t,t1),x1,if(lt(t,t2),x2,...))`
expression. Plan §13 #12 caps depth at 200 levels; the §2.4 commentary
says "超长视频按 0.5s 步长降采样". This check measures the actual
ffmpeg runtime limit on the target build.

### Methodology

Build expressions of progressively deeper nesting and feed them to
`crop=W:H:expr:0` against `sample.mp4`:

```python
def build_expr(n: int) -> str:
    expr = '0'
    for i in range(n):
        t = (i + 1) * 0.5
        x = 100 + (i % 200)
        expr = f'if(lt(t\\,{t})\\,{x}\\,{expr})'
    return expr
```

Run for `n ∈ {80..104}` and record success/failure of `subprocess.run`:

```powershell
py -3.11 probe_depth.py
```

### Results (ffmpeg 8.1-full_build-www.gyan.dev, Windows 10)

| `n` (`if(...)` levels) | Verdict |
| ----------------------:|:------- |
| 80 .. 98               | OK      |
| **99**                 | **FAIL** — `Missing ')' or too many args` from `[Eval @ ...]` (recursion-depth cutoff inside `libavutil/eval.c`) |
| 100 .. 104             | FAIL    |

Concrete failure shape at `n=99`:

```
[Parsed_crop_0 @ ...] [Eval @ ...] Missing ')' or too many args in
'if(lt(t,1.5),102,if(lt(t,1.0),101,if(lt(t,0.5),100,0))))))))))))))…'
```

The cutoff is **n=98** (passes) → n=99 (fails). This is *much* lower
than the plan's tentative 200-level estimate (red-line §13 #12). On
older ffmpeg builds the limit is sometimes documented as 100; ffmpeg
8.1 on this Windows build appears to enforce ≤98.

### Decision

**Lower the cap from 200 to 95** in `mediapost_recompose.py` (Phase 3,
§6.2 step 4). 95 leaves a safety margin under the measured 98-level
ceiling without forcing aggressive downsampling on typical clips:

- Per §6.2 the expression is built **per scene segment**, not over the
  whole video. With `scene_threshold=0.4` typical footage cuts every
  3–10 seconds, so per-segment frame counts at `recompose_fps=2.0`
  rarely exceed 20 (well under 95).
- Long unbroken talking-head segments stay safe by an automatic
  per-segment downsample to ≤95 sample points: the recompose builder
  picks every `ceil(N / 95)`-th frame and lets ffmpeg interpolate
  linearly between them. The dropped points are still fed through the
  EMA smoother first so accuracy is unaffected.
- The Settings tab can re-tune `recompose_fps` (1.0 / 2.0 / 3.0) for
  users who want denser sampling, but the per-segment 95-cap stays
  fixed so the expression never trips ffmpeg's parser.

A regression test in Phase 3 will hard-code:

```python
# tests/test_recompose.py (Phase 3 deliverable)
def test_crop_expression_caps_at_95_levels():
    expr = build_crop_x_expression(time_centers_with_500_points)
    assert _max_if_depth(expr) <= 95
```

so that a future refactor can't silently push the depth back up.

```powershell

  -f lavfi -i "color=green:size=320x240:duration=2,format=yuv420p" `
```
